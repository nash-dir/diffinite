## 괜찮았던 것

### 1. 기반 알고리즘 선택

Winnowing (Schleimer 2003)은 MOSS의 기반이고, 학계와 법정에서 20년간 검증된 알고리즘이다. 이 선택은 맞다. 직접 구현한 것도 합리적이다 — 핵심 알고리즘을 블랙박스로 쓰면 포렌식 도구로서의 신뢰성이 떨어진다.

### 2. 인프라 품질

`parser.py`(5-state FSM), `collector.py`(2-phase matching), `differ.py`(context folding) 등 인프라 계층의 엔지니어링 품질은 높다. 성능 최적화도 실측 기반(4.4×, 1824×)으로 이루어졌다. 이 부분은 논문이든 제품이든 그대로 쓸 수 있다.

### 3. 테스트 투자

253개 테스트는 이 규모의 프로젝트에서 양호하다. 특히 `test_parser.py`처럼 경계 조건(문자열 내 `#`, 트리플 쿼트, 백틱)을 꼼꼼히 테스트한 것은 프로토타입 단계를 넘어서려는 좋은 습관이다.

### 4. 모듈 분리

`cli → pipeline → collector/parser/differ/fingerprint/deep_compare → evidence → pdf_gen` 의존성 구조는 깨끗하다. 각 모듈의 책임이 명확하고 순환 의존이 없다. 이것은 리팩토링이 용이한 구조이며, `paper` 브랜치에서 `evidence.py`만 교체할 수 있는 이유다.

### 5. 이 대화에서의 자기 인식

"내가 뭘 요구하고 있는지 모르면서 push the blue button"이라는 인식 자체가 가장 중요한 깨달음이다. 대부분의 프로젝트가 이 인식 없이 끝까지 가버린다.

## 고쳐야 할 것

### 1. 기능 추가는 빨랐지만 검증은 없었다

dev 브랜치의 타임라인을 보면:

```
3/13: Winnowing 기본 구현
3/14: 6채널 증거, 2단계 분류, AFC, IDEX, 언어 레지스트리
3/15: TDD 8-stage, grid search, annotation 12개 모듈
3/16: Phase 2 재구현 6건 + 문서 4건 + 코드 품질 2건
```

**3일 만에 1168줄짜리 evidence.py가 만들어졌다.** 하지만 그 어느 시점에서도 "6채널이 3채널보다 낫다"는 실험은 없었다. "2단계 분류가 1단계보다 낫다"는 비교도 없었다. 기능이 추가될 때마다 프로젝트는 더 복잡해졌지만, 그 복잡성이 정당화되었는지는 한 번도 검증되지 않았다.

**교훈**: 채널 하나 추가하기 전에 "이것 없이 AUC가 얼마인지" 먼저 측정해야 한다.

### 2. 복잡성 자체가 목표가 되었다

evidence.py의 분류 체계를 보자:

- 7개 분류 라벨 (DIRECT_COPY, SSO_COPYING, OBFUSCATED_CLONE, DOMAIN_CONVERGENCE, SUSPICIOUS_COPY, SUSPICIOUS_SSO, INCONCLUSIVE)
- 2개 프로파일 (industrial, academic)
- 18개 임계값 파라미터
- 3단계 AFC 파이프라인
- 5개 법리 패턴
- 8개 IDEX 파라미터

이 모든 것이 646쌍의 데이터 위에 세워져 있다. **데이터에 비해 모델이 지나치게 복잡하다.** 7-class 분류기를 646개 샘플로 튜닝하면 클래스당 평균 92개인데, 여기에 18개 임계값을 grid search한다. 이것은 시스템이 데이터를 "외워버리는" 구조다. 단순한 3-class 분류기 (COPYING / INCONCLUSIVE / INDEPENDENT)와 비교해본 적이 있는가?

**교훈**: "법정에서 인상적으로 보이는 7-class 분류"보다 "통계적으로 방어 가능한 3-class 분류"가 실제 법정에서 더 강하다.

### 3. 주석이 논증을 대신하지 않는다

12개 모듈에 "시니어 아키텍트 수준 주석"을 추가하는 데 상당한 시간을 투입했다. 예를 들어:

```python
# 84K 조합 grid search 최적화 (zero-FP 목표)
_DC_RAW_MIN = 0.65
```

이 주석은 "85K가 아니라 84K인 이유"를 설명하는 것이 아니라, **"84K"라는 숫자로 엄밀함의 인상을 준다.** 진짜 필요한 주석은:

```python
# 5-fold CV mean=0.63 (std=0.04, 95% CI [0.56, 0.71])
# see report/experiments/E3_threshold_cv.md
```

현재 주석의 상당수가 "이 코드가 무엇을 하는지" 설명하면서 "이 값이 왜 이 값인지"는 설명하지 않는다.

**교훈**: 주석은 **결정의 근거**를 기록하는 것이지, 코드의 동작을 반복 서술하는 것이 아니다.

### 4. `git filter-repo` 후 복원은 재검토의 기회였다

Phase 2 고도화 6건이 유실되었을 때, 즉시 "전량 복원"을 요청했다. 그런데 유실된 6건 중:

- `TOKEN_RE` 통합: 합리적 리팩토링 → **복원 정당**
- `autojunk` CLI: 표준 파라미터 노출 → **복원 정당**
- `ClassificationThresholds` 18필드: ad-hoc 값을 frozen dataclass로 포장 → **복원이 오히려 유해**
- `IDEXThresholds` 8필드: "직관적 추정"의 형식화 → **복원이 오히려 유해**
- JS 테스트 3건: 순수 테스트 → **복원 정당**
- 메모리 제한: 순수 엔지니어링 → **복원 정당**

6건 중 2건은 **복원하지 않는 것이 더 나았다.** 매직넘버를 dataclass로 감싸면 "정식 API"의 외양을 갖추게 되어, 나중에 "이것은 정당한 파라미터"라는 착각을 유발한다. 실제로 바로 다음 단계에서 `_get_thresholds(profile)` 함수가 생겼고, 이제 이 ad-hoc 값들은 시스템 전체에서 "공식 임계값"으로 참조된다.

**교훈**: 유실은 때때로 정리의 기회다. "원래 있었으니 복원"이 아니라 "이것이 있어야 하는가?"를 먼저.

### 5. 보고서 작성이 실행보다 많아졌다

파일 목록을 보면:

```
report/audit_260315.md
report/todo_2603152026.md
report/todo_260316.md
report/lessons_learned.md  ← 이 문서까지 포함하면 4번째
doc/progress.md
```

보고서가 5개이고, 코드 변경은 실질적으로 `evidence.py`의 dict→dataclass 마이그레이션과 몇 개 파라미터 전파가 전부다. **보고서 작성에 투입된 시간이 실험 설계나 검증에 투입되었으면** 지금쯤 5-fold CV 결과가 나와 있을 것이다.

**교훈**: 문서화는 가치 있지만, "무엇을 할 것인지 기술하는 것"이 "실제로 하는 것"을 대체해서는 안 된다.

### 6. "법리 분석" 기능의 위험성

`analyze_legal_defense_pattern()`은 5개 법리 패턴을 출력한다: CLEAN_ROOM_PROBABLE, LITERAL_COPYING 등. 이 함수의 출력이 누군가의 법적 분쟁에서 인용되면?

현재 이 분류 기준은 **검증되지 않은 8개 매직넘버**에 기반한다. 면책 조항이 있지만, "Diffinite가 CLEAN_ROOM_PROBABLE이라고 판정했다"라는 문장이 법정 서류에 등장하는 순간, 상대 측 전문가 증인이 "이 0.20이라는 값의 근거가 무엇입니까?"라고 물을 것이다. 대답할 수 없다.

**교훈**: 법리 분석 기능은 "데이터에 의해 정당화될 때까지" 실험적(experimental) 태그를 달아야 한다. 또는 점수만 제시하고, 패턴 분류는 사용자에게 맡겨야 한다.
