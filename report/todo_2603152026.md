## 1. 분류 임계값 외부화

### 문제 진단

`evidence.py`에 하드코딩된 분류 임계값이 **3개 레이어**에 분산되어 있다:

```
_CLASSIFICATION_PROFILES     → 12개 키 × 2개 프로파일 = 24개 매직 넘버
_AFC_SSO_DECL_MIN, _AFC_SSO_GAP_MIN  → AFC 전용 2개
_RELAXED_DC_RAW_MIN 등       → Relaxed 전용 3개
_SSO_NORM_RAW_RATIO          → 비율 임계값 1개
```

총 **30개 매직 넘버**가 모듈 상수로 흩어져 있고, 도출 근거가 주석에만 존재.
프로파일 추가(예: `--legal-idex`) 시 새 dict 항목 + 모듈 상수 + 주석을 모두 동기화해야 하므로 실수 위험이 높다.

### 해결 전략

**방안 A — `dataclass` 구조화 (권장)**

```python
@dataclass(frozen=True)
class ClassificationThresholds:
    """분류 임계값 사양. frozen으로 런타임 변경 방지."""
    dc_raw_min: float = 0.65
    dc_ident_min: float = 0.50
    sso_raw_max: float = 0.20
    sso_decl_min: float = 0.60
    sso_gap_min: float = 0.30
    sso_ast_min: float = 0.25
    obc_raw_max: float = 0.15
    obc_ident_max: float = 0.30
    obc_ast_min: float = 0.30
    conv_ident_min: float = 0.20
    conv_decl_max: float = 0.40
    conv_raw_max: float = 0.20
    # AFC 오버라이드
    afc_sso_decl_min: float = 0.75
    afc_sso_gap_min: float = 0.35
    # Relaxed
    relaxed_dc_raw_min: float = 0.50
    relaxed_sso_decl_min: float = 0.50
    relaxed_sso_gap_min: float = 0.25
    # 비율
    sso_norm_raw_ratio: float = 1.2

INDUSTRIAL = ClassificationThresholds()
ACADEMIC = ClassificationThresholds(
    dc_raw_min=0.70, dc_ident_min=0.60, ...
)
```

장점:
- 타입 체크 가능, IDE 자동완성 지원
- 새 프로파일은 `ClassificationThresholds(...)` 한 줄로 생성
- `frozen=True`로 런타임 변경 방지

**방안 B — YAML/JSON 외부화**

프로파일을 `config/profiles/industrial.yaml` 등으로 분리.
장점: 코드 수정 없이 프로파일 추가 가능.
단점: 파일 로딩 로직 추가, Python 타입 안전성 상실.

### 구현 방향

1. `models.py`에 `ClassificationThresholds` dataclass 추가
2. `evidence.py`의 `_CLASSIFICATION_PROFILES` dict를 dataclass 인스턴스로 교체
3. `_classify_strict()`, `_classify_relaxed()` 함수가 dataclass 필드를 참조하도록 수정
4. 모듈 상수 (`_DC_RAW_MIN` 등 레거시 alias) 제거
5. 기존 테스트 전체 통과 확인

### 프롬프트

```
evidence.py의 분류 임계값을 dataclass로 구조화해 줘.

현재 상태:
- _CLASSIFICATION_PROFILES dict에 24개 매직 넘버가 하드코딩
- _AFC_*, _RELAXED_*, _SSO_NORM_RAW_RATIO가 별도 모듈 상수
```

## 2. 법리 분석 임계값 교정

### 문제 진단

`evidence.py`의 `_compute_idex_delta()` 및 관련 법리 분석 임계값은
**직관적 추정**으로 설정되어 있다. 다른 모든 임계값은 646쌍 코퍼스에서
grid search로 교정되었으므로, 이 부분만 검증 없이 남아 있다.

해당 임계값:
- `_IDEX_EXPRESSION_THRESHOLD` — "표현 수준 유사도" 판별선
- `_IDEX_IDEA_THRESHOLD` — "아이디어 수준 유사도" 판별선
- industrial/academic K,W 프로파일 값

2. **ROC AUC 기반 최적 컷오프**: Youden's J statistic으로 최적 임계값 도출
3. **교차 검증**: 코퍼스를 5-fold로 나눠 임계값 안정성 확인
4. **결과 반영**: 도출된 값으로 임계값 교체, 주석에 도출 근거 기록

### 구현 방향

1. `TDD/corpus/` 하위에 `idex_threshold_calibration.py` 스크립트 작성
2. 기존 코퍼스 파일 쌍에 대해 IDEX 점수 수집
3. 양성/음성별 분포 히스토그램 + ROC 곡선 생성
4. 최적 임계값 → `evidence.py` 업데이트
5. 테스트 추가: `test_idex_thresholds.py`

### 프롬프트

## 3. 역 인덱스 메모리 제한

### 문제 진단

`deep_compare.build_inverted_index()`는 B-파일의 **모든 해시를 메모리에 적재**한다.
Winnowing의 해시 수는 파일 길이에 비례하므로:

```
예: 10,000 파일 × 평균 500 해시 = 5,000,000 엔트리
    해시(8B) + file_id 참조(~50B) ≈ 290MB
```

현재 규모(수백 파일)에서는 문제없지만, 모노레포 비교(10만+ 파일) 시 OOM 위험.

### 해결 전략

**단기 (안전장치):**
- `build_inverted_index()`에 `max_entries` 파라미터 추가
- 초과 시 경고 로그 + graceful degradation (나이브 비교로 폴백)

**장기 (근사 기법):**
- **MinHash + LSH**: O(1) 근사 Jaccard. `datasketch` 라이브러리 활용
- 해시 수 대신 MinHash 시그니처(128~256차원)만 저장 → 메모리 90%+ 절감
- 정확도 트레이드오프: Jaccard 오차 ±3% 이내 (128-perm 기준)

### 구현 방향

**단기:**
1. `build_inverted_index()`에 `max_entries` 파라미터(기본 10M) 추가
2. 초과 시 `logger.warning()` + `set` 대신 샘플링된 subset 사용
3. CLI에 `--max-index-entries` 옵션 추가

**장기:**
1. `pip install datasketch` 의존 추가 (선택적)
2. `deep_compare.py`에 `_build_minhash_index()` 대안 함수 구현
3. `--approximate` CLI 플래그로 전환 가능하게 설계
4. 정확도 검증: 기존 코퍼스에서 exact vs. approximate 비교

### 프롬프트

```
deep_compare.py의 역 인덱스에 메모리 안전장치를 추가해 줘.

현재 상태:
- build_inverted_index()가 모든 해시를 메모리에 적재
- 대규모 코퍼스에서 OOM 위험

단기 목표:
1. max_entries 파라미터 추가 (기본 10,000,000)
2. 초과 시 warning 로그 + graceful degradation
3. CLI에 --max-index-entries 옵션 추가
4. 테스트 추가 및 전체 테스트 통과 확인

장기 목표(별도 작업):
- MinHash + LSH 기반 근사 인덱스 (datasketch 활용)

//turbo-all
```

## 4. JS 템플릿 리터럴 파서 보완

### 문제 진단

`parser.py`의 상태 머신이 백틱(`` ` ``)을 단일 문자 구분자로 처리하여,
JavaScript 템플릿 리터럴의 **내부 표현식 중첩을 무시**한다:

```javascript
const msg = `Hello ${user.name}, you have ${getCount()} items`;
//                   ^^^^^^^^^^^^^^^^       ^^^^^^^^^^^^^
//                   이 영역의 코드가 문자열로 취급됨
```

`${...}` 내부에 `//` 주석이나 `/* */`가 있으면 오탐 가능:

```javascript
const x = `result: ${
  value /* important comment */ + 1
}`;
// → "/* important comment */"가 주석으로 오인될 수 있음
```

### 해결 전략

상태 머신에 `IN_TEMPLATE_LITERAL` 상태를 추가하고,
`${` 를 만나면 depth 카운터를 증가시키며 `CODE` 상태로 전환.
`}`를 만나면 depth를 감소시키고, 0이 되면 `IN_TEMPLATE_LITERAL`로 복귀.

```
CODE → ` → IN_TEMPLATE_LITERAL
IN_TEMPLATE_LITERAL → ${ → CODE (depth++)
CODE → } (depth > 0) → IN_TEMPLATE_LITERAL (depth--)
IN_TEMPLATE_LITERAL → ` → CODE
```

### 구현 방향

1. `_State` enum에 `IN_TEMPLATE_LITERAL` 추가
2. `_strip_2pass()`에서 백틱 진입 시 `IN_TEMPLATE_LITERAL` 사용
3. `${` 감지 → `CODE` 전환 + `template_depth` 카운터
4. `}` 감지 시 depth 체크 → 0이면 `IN_TEMPLATE_LITERAL` 복귀
5. 테스트 추가: 중첩 템플릿 리터럴, 내부 주석, 내부 문자열

### 프롬프트

```
parser.py의 상태 머신에 JavaScript 템플릿 리터럴 지원을 추가해 줘.

현재 상태:
- 백틱(`)을 단일 문자 구분자로 처리
- ${...} 내부의 코드가 문자열로 취급됨
- 내부 주석/문자열이 오탐될 수 있음

목표:
1. _State에 IN_TEMPLATE_LITERAL 추가
2. ${} 중첩을 depth 카운터로 추적
3. 중첩 구조: ` → IN_TEMPLATE_LITERAL → ${ → CODE → } → IN_TEMPLATE_LITERAL → `
4. 테스트 추가:
   - 기본 템플릿 리터럴
   - 내부 주석 포함 (// 및 /* */)
   - 내부 문자열 포함
   - 중첩 템플릿 리터럴
5. 기존 테스트 전체 통과 확인

//turbo-all
```

## 5. `_TOKEN_RE` 중복 통합

### 문제 진단

**완전히 동일한 정규식**이 두 모듈에 독립 정의되어 있다:

```python
# fingerprint.py:24
_TOKEN_RE = re.compile(r"[A-Za-z_]\w*|[0-9]+(?:\.[0-9]+)?|[^\s]")

# evidence.py:34
_TOKEN_RE = re.compile(r"[A-Za-z_]\w*|[0-9]+(?:\.[0-9]+)?|[^\s]")
```

두 모듈이 동일한 토크나이저를 사용해야 채널 간 일관성이 보장되는데,
현재 구조에서는 **한쪽만 수정하면 미묘한 불일치가 발생**할 수 있다.

### 해결 전략

`fingerprint.py`에 정의된 `_TOKEN_RE`를 canonical로 삼고,
`evidence.py`에서 import하여 사용. 또는 `languages/_spec.py`의
공통 유틸리티로 이동.

### 구현 방향

1. `fingerprint.py`의 `_TOKEN_RE`를 public으로 전환 (`TOKEN_RE`)
2. `evidence.py`에서 `from diffinite.fingerprint import TOKEN_RE` 사용
3. 기존 `evidence.py`의 `_TOKEN_RE` 정의 제거
4. 테스트 전체 통과 확인

**대안**: `_COMMON_KEYWORDS`도 같은 패턴이므로 함께 정리.
현재 `evidence.py`가 이미 `from diffinite.fingerprint import _COMMON_KEYWORDS`를
하고 있으므로, `_TOKEN_RE`도 같은 경로로 통합하면 자연스럽다.

### 프롬프트

```
fingerprint.py와 evidence.py에 중복 정의된 _TOKEN_RE를 통합해 줘.

현재 상태:
- 동일한 정규식이 fingerprint.py:24와 evidence.py:34에 각각 정의
- evidence.py는 이미 fingerprint.py에서 _COMMON_KEYWORDS를 import 중

목표:
1. fingerprint.py의 _TOKEN_RE를 canonical 소스로 사용
2. evidence.py에서 import하여 중복 제거
3. 기존 테스트 전체 통과 확인

//turbo-all
```

## 6. `autojunk` CLI 노출

### 문제 진단

`differ.py`의 `compute_diff()`가 `difflib.SequenceMatcher(autojunk=True)`를
기본으로 사용한다. `autojunk=True`는 200줄 이상의 파일에서 **반복 패턴을
'정크'로 판별**하여 매칭에서 제외한다.

포렌식 관점에서의 문제:
- import 문, 라이선스 헤더 등이 junk 판정 → 유사도가 실제보다 높게 산출
- 분석 재현성 저하: autojunk의 내부 휴리스틱이 파일 길이에 따라 달라짐
- 법정에서 "도구가 일부 코드를 무시했다"는 반증에 취약

### 해결 전략

`compute_diff()`에 `autojunk` 파라미터를 추가하고,
CLI에서 `--no-autojunk` 플래그로 비활성화할 수 있게 한다.
기본값은 하위 호환을 위해 `True` (현재 동작 유지).

### 구현 방향

1. `differ.compute_diff()`에 `autojunk: bool = True` 파라미터 추가
2. `SequenceMatcher(None, ..., autojunk=autojunk)` 전달
3. `cli.py`에 `--no-autojunk` 플래그 추가
4. `pipeline.run_pipeline()`에서 CLI 인자를 `compute_diff()`에 전달
5. `models.AnalysisMetadata`에 `autojunk: bool` 필드 추가 (보고서 투명성)
6. 테스트 추가: autojunk=True/False 비교 케이스

### 프롬프트

```
differ.py의 autojunk 옵션을 CLI로 노출해 줘.

현재 상태:
- compute_diff()가 SequenceMatcher(autojunk=True) 고정 사용
- 포렌식 목적에서 autojunk 비활성화가 필요할 수 있음

목표:
1. compute_diff()에 autojunk 파라미터 추가 (기본 True)
2. CLI에 --no-autojunk 플래그 추가
3. pipeline.run_pipeline()에서 전달
4. AnalysisMetadata에 autojunk 필드 추가
5. 테스트 추가: autojunk=True vs False 비교
6. 기존 테스트 전체 통과 확인

//turbo-all
```
