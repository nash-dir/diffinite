## 2. 코드 품질 진단

### 2.1 ✅ 강점

**아키텍처**
- 모놀리식(구 `diffinite.py` 36KB) → 11개 단일 책임 모듈로 분리 완료
- 의존성이 단방향이므로 개별 모듈 교체/확장이 용이
- `models.py` 분리로 순환 참조 원천 차단

**타입 안정성**
- 공개 함수 타입 힌트 커버리지 **99%** (106개 인자 중 105개)
- `dataclass` 기반 모델로 런타임 타입 일관성 보장

**알고리즘 다층화**
- 4단계 분석 체계 (token → normalized → AST → PDG)
- 5채널 증거 스코어링으로 법적 증거의 다각적 입증 가능
- Winnowing 알고리즘은 학술적으로 검증된 표준 기법 (Schleimer et al., 2003)

**테스트 인프라**
- 178개 테스트, 모든 공개 함수 커버
- 실데이터 통합 테스트 (SQLite 250K-line, AOSP Java)
- 성능 벤치마크 내장 (sqlite3.c strip < 2s, diff < 30s)

### 2.2 ⚠️ 주의 사항

| 영역 | 현재 상태 | 위험도 | 설명 |
|---|---|---|---|
| **에러 핸들링** | 부분적 | 🟡 중간 | `pipeline.py`에 13개 로깅이 집중. `fingerprint.py`, `evidence.py`는 logging 0개 |
| **PDF 렌더링** | `xhtml2pdf` | 🔴 높음 | CSS 지원 빈약, 대형 파일 메모리 이슈 가능 |
| **CLI UX** | 최소 | 🟡 중간 | Progress bar 없음, 에러 시 사용자 피드백 부족 |
| **PDG 분석 범위** | 함수 내부만 | 🟡 중간 | Cross-function/cross-file 의존성 미분석 |
| **대규모 확장성** | N×M 비교 | 🟡 중간 | Inverted index 최적화 있지만 수천 파일 미검증 |

### 2.3 ❌ 기술 부채

1. **`pdf_gen.py` 비대** — 634줄로 전체의 19%. CSS, HTML 템플릿, PDF 변환, 병합, Bates 스탬프가 한 파일에 혼재
2. **하드코딩된 상수** — `CONTEXT_LINES=3`, `FUZZY_THRESHOLD=60` 등이 모듈 내부에 흩어져 있음. 설정 파일 미지원
3. **`deep_compare.py` 복잡도** — 417줄에 멀티프로세싱 + inverted index + 다채널 스코어링이 한 함수(`run_deep_compare`)에 밀집
4. **Example 데이터 관리** — `setup_example_sqlite.py`가 외부 URL에서 다운로드. CI에서 불안정할 수 있음

## 5. 총평

Diffinite v0.3.0은 **"학술 알고리즘을 갖춘 실용 도구의 초기 완성형"**이다.

코드 아키텍처는 상용 수준의 모듈 분리와 타입 힌트 커버리지(99%)를 갖추고 있으며, 4단계 분석(token → normalized → AST → PDG)과 5채널 증거 체계는 학술 도구에서도 보기 드문 다층 구조다. 178개 테스트로 회귀 방지 안전망이 견고하다.

가장 시급한 개선점은 **PDF 렌더링 안정성**과 **대규모 레포 확장성**이다. 현재 `xhtml2pdf`의 CSS 한계는 결과물의 신뢰도를 떨어뜨리며, N×M 전수비교는 파일 수가 500개를 넘으면 실용적이지 않다. 이 두 가지를 해결하면 PyPI 배포 즉시 niche 도구로서 경쟁력을 갖출 수 있다.

**단기 목표**: P1(PDF) + P2(CLI UX) + E1(JSON 출력) 해결 → **v0.4.0**으로 상용 배포 가능 수준 달성
